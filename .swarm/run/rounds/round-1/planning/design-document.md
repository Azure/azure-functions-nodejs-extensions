## Summary
`azure-functions-nodejs-extensions-servicebus/src/servicebus/ServiceBusMessageActions.ts` currently hard-codes `grpc.credentials.createInsecure()`, and `src/grpcClientFactory.ts` still defaults to insecure credentials. This should be hardened, but a repo-only flip to TLS would break current Azure Functions host compatibility.

## Findings
- **Is a fix necessary?** Yes. The package should stop blindly opting into insecure transport. This repo can harden credential selection and block unsupported insecure remote endpoints, but full end-to-end TLS requires host/runtime support outside this repo.
- **Historical context**: The behavior was inherited from the Azure Functions worker architecture, which still launches workers with `--functions-uri http://localhost/...` in `Azure/azure-functions-host/src/WebJobs.Script.Grpc/Rpc/RpcWorkerContext.cs`, and the upstream Node worker still uses `grpc.credentials.createInsecure()` in `Azure/azure-functions-nodejs-worker/src/GrpcClient.ts`. In this repo, `createInsecure()` has existed since the initial Service Bus extension commit (`08cbd57`). The original constructor allowed injected credentials, so the loopback/plaintext choice was intentional for local host comms; the later singleton refactor (`baf8d64`) hard-coded insecure credentials again, which turned this into a missed hardening gap.
- **Regression risk**: Unconditionally switching to TLS here would likely break every current settlement call. The safe repo-local change is to prefer `functions-uri`, preserve the endpoint security intent, keep current loopback/http behavior for today’s supported host, and refuse insecure non-loopback endpoints instead of silently allowing them.
- **Customer contract / breaking change**: No public `IServiceBusMessageActions` or settlement API change is needed. Internal helper signatures can change, but the customer-facing contract should remain stable. Any true TLS rollout is a separate cross-repo/runtime contract change, not a package-breaking change in this repo alone.
- **Testing status**: The package has about 237 unit tests, including 40 in `ServiceBusMessageActions.test.ts`, but there is no dedicated coverage for transport credential policy, and one existing test explicitly codifies insecure-by-default behavior. `npm test` could not be executed in this workspace because dev dependencies are absent (`ts-node: not found`), so the updated suite needs validation in CI or another dependency-enabled environment.

## Repo-local implementation
1. Add an internal connection/credential resolver (new util or extended `GrpcUriBuilder`) that prefers `--functions-uri`, falls back to `--host`/`--port`, and returns `address`, `grpcMaxMessageLength`, and explicit channel credentials/security metadata.
2. Update `ServiceBusMessageActions` to use that resolver and remove the hard-coded `createInsecure()` call; also remove the insecure default from `createGrpcClient` so future call sites must choose credentials intentionally.
3. Expand unit tests around URI parsing, credential selection, loopback fallback, and unsupported insecure remote endpoints.
